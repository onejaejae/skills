# Phase 2: Domain Deep Dive (상세 가이드)

**THE MOST IMPORTANT PHASE.** 도메인을 가르치는 것이 목표 — 용어 나열이 아니라 WHY를 설명.
개발자가 이 문서를 읽고 좋은 제품 결정을 내릴 수 있어야 한다.

Quick analysis(단일 화면)에서는 도메인 핵심을 Phase 3에 인라인. 멀티 스크린은 반드시 full Phase 2.

## Step 1: Domain Context

```markdown
### This Domain in a Nutshell

**Industry:** [e.g., 의료 연구 / HR / 이커머스 / 교육]
**Core problem this domain solves:** [e.g., "임상 연구에서 '고혈압 환자'를 정의하려면, 수천 개의 의료 코드 중 정확한 코드를 선별해야 한다."]
**Why it matters:** [e.g., "연구의 재현성과 정확성은 코드셋 정의의 품질에 직결된다."]
**Key industry standards/concepts:** [e.g., "ICD-10, LOINC, ATC 등 국제 표준 코드 체계"]
```

**RULE:** 도메인 지식 제로인 신규 팀원에게 가르치듯 작성. 구체적 예시 필수. 전문 용어 설명.

## Step 1.5: End-to-End Domain Workflow (Standard/Deep 필수)

개별 용어 전에 **도메인 전체 워크플로우**를 먼저 설명. 각 개념의 존재 이유가 자명해진다.

```markdown
### [도메인명] 전체 흐름 (배경지식)

이 제품을 이해하려면, 먼저 **[도메인]이 어떤 과정으로 이루어지는지** 알아야 합니다.

[1] [첫 번째 단계]
    설명: ...
        ↓
[2] [두 번째 단계]
    설명: ...
        ↓
...
[N] [마지막 단계]

**이 제품이 자동화하는 구간**: [X]~[Y]

**왜 이 과정이 어려운가 (제품의 존재 이유):**
| 단계 | 어려운 이유 | 이 제품이 돕는 방식 |
|------|-----------|-------------------|
| [단계 N] | [구체적 난이도] | [제품의 해결 방식] |

**이 제품의 도메인 내 위치:**
[여러 유형/접근법이 있다면, 이 제품이 어떤 유형에 해당하는지]
```

**RULE:** 워크플로우는 **도메인 자체의 업무 프로세스**여야 한다.
- X: "코드셋 생성 → 검증 → 저장" (제품 기능)
- O: "연구 질문 → 가설 → 연구 설계 → 코호트 → 변수 → 데이터 수집 → 통계 → 논문" (도메인 워크플로우)

**Each concept should map back:** Phase 3에서 `> 도메인 워크플로우 [N] — "무엇을, 어떻게?"` 형태로 연결.

## Step 2: Domain Glossary

UI에서 모든 도메인 용어 추출: 네비게이션, 타이틀, 프레임명, 필드 레이블, 테이블 헤더, 버튼, 상태 배지, 드롭다운, 탭, 브레드크럼.

**3계층 분리:**

1. **핵심 도메인 용어** — 제품 이해에 필수 (예: 코드셋, 연구코호트)
2. **업계 표준 용어** — UI에 등장하는 전문 용어 (예: MACE, SGLT-2). "쉬운 설명" + "이 제품에서의 맥락" 컬럼 포함
3. **코드/분류 체계** — 표준 코드 시스템 (예: ICD-10). 용도와 상호 관계 설명

각 계층 **별도 표**. 2, 3번은 도메인에 따라 생략 가능.

**형식 — MEANING, not translation:**

| Term | What It Means (In Plain Language) | Why It Exists | Related Terms |
|------|-----------------------------------|---------------|---------------|
| 연차 | 근속연수에 따라 부여되는 유급 휴가. 한국 근로기준법 의무. 1년 미만 월 1일, 이후 연 15일. | 법적 의무이자 직원 복지 기본 | 병가, 경조사 |

- BAD: `| 코드셋 | Code Set | 코드의 집합 |`
- GOOD: `| 코드셋 | 특정 연구 목적에 맞게 선별한 의료 코드의 집합. 코드 누락→연구 대상 빠짐, 잘못 포함→결과 오염. | 연구 정확성/재현성 보장 | UDM, 진단코드 |`

## Step 2.5: Core Concept Deep Teaching (Standard/Deep 필수)

용어집은 quick reference. 핵심 개념(3~5개)은 **독립 섹션으로 깊이 설명**.

**선정 기준:** UI에서 반복 등장하며, 이해 없이는 "왜"를 파악할 수 없는 개념.

**각 개념에 필수 6가지 요소:**

```text
#### [개념명]이란?

**한 줄 정의:** [전문 용어 없이]

**비유:** [일상 경험에 비유. 예: "Index date = 스톱워치의 '시작' 버튼을 누르는 시점"]

**왜 중요한가?**
[이 개념이 없거나 잘못되면 무슨 일이 생기는지]

**예시:**
[관통 예시(running example) 시나리오로 설명]

[ASCII 다이어그램 — 시간축, 흐름, 구조 등 시각화]

**이 제품에서:** [UI의 어디에, 어떤 형태로 나타나는지]
```

**RULE: 관통 예시 (Running Example)** — 하나의 구체적 시나리오를 선정하여 **모든** 핵심 개념을 동일 시나리오로 설명. 예시가 바뀌면 독자가 매번 새 맥락을 학습해야 한다.

## Step 3: Domain Entities and How They Relate

UI에서 엔티티(명사)와 관계를 추출. **비즈니스 의미** 중심:

```markdown
### Why does [Entity A] exist?
[Entity A]는 [비즈니스 맥락] 때문에 필요하다.

### How do entities connect?
연구자가 새 연구를 시작하면:
1. [Entity A] 정의 — 연구 범위 결정
2. [Entity A] 안에 [Entity B] 추가 — 표준 코드 참조
3. [Entity C]가 B의 상태 관리 — 코드 검증 워크플로우
```

관계 요약:
```text
Employee 1──N LeaveRequest    "직원은 여러 휴가를 신청"
Department 1──N Employee      "부서에 여러 직원"
```

## Step 4: State Machines and Workflows

상태 배지가 있는 엔티티마다 상태 전이 매핑 + 비즈니스 이유:

```markdown
신규 ──(검토요청)──> 검토중 ──(승인)──> 확인됨
                              ──(반려)──> 수정필요

**왜 이런 워크플로우가 있는가:**
잘못된 코드 하나가 연구 결과 전체를 무효화할 수 있어
코드 추가는 반드시 검증 과정을 거쳐야 한다.
```

## Step 5: Domain Knowledge the Design Implies

보이는 것 너머를 추론:

```markdown
### Design이 암시하는 도메인 지식

- **4개 카테고리 분리 (진단/약품/검사/수술):** 각 카테고리마다 다른 표준 코드 체계를 사용 (진단=ICD-10, 약품=ATC 등)
- **약품 테이블의 컬럼이 가장 많은 이유:** 같은 약이라도 용량, 투여경로, 빈도, 제형에 따라 완전히 다른 처방
```

## Phase 2 Completion Checklist (자가 검증)

Phase 2를 마친 후 점검. 하나라도 No면 보완:

**도메인 워크플로우:**
- [ ] 제품 기능이 아닌 도메인 업무 프로세스로 서술되었는가?
- [ ] "왜 이 과정이 어려운가" 테이블이 있는가?
- [ ] 이 제품이 워크플로우의 어느 구간을 자동화하는지 명시했는가?

**용어집:**
- [ ] 모든 화면의 도메인 용어가 포함되었는가?
- [ ] 계층별 분리(핵심/업계 표준/코드 체계)되었는가?
- [ ] 각 용어에 비즈니스 맥락이 있는가? (단순 번역 금지)

**핵심 개념 교육:**
- [ ] 3~5개 핵심 개념이 독립 섹션으로 설명되었는가?
- [ ] 6요소(정의/비유/중요성/예시/다이어그램/제품 연결)를 갖추었는가?
- [ ] 관통 예시가 일관되게 사용되고 있는가?

**엔티티 & 상태:**
- [ ] 각 엔티티의 비즈니스 존재 이유가 설명되었는가?
- [ ] 상태 전이마다 워크플로우 존재 이유가 있는가?

**전체:**
- [ ] 도메인 지식 제로인 사람이 "왜 이렇게 만들어졌는지" 이해할 수 있는가?
