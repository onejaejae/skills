# 출력 형식 가이드

**출력물의 정체성: "제품 & 도메인 가이드"** — Figma Analysis Spec이 아니라, 도메인과 제품을 이해시키는 가이드 문서.

## Quick Analysis (conversation)

단일 화면, 빠른 개요:
- 도메인 맥락 (1 paragraph)
- 제품 big picture
- 핵심 파이프라인 요약
- 주요 도메인 개념 인라인 설명

## Writing Rules (모든 depth에 적용)

### 섹션 도입부 (MANDATORY)

모든 주요 섹션(##)은 반드시 1-2문장의 블록인용(>)으로 시작:
- 이 섹션은 **무엇**에 대한 것인가?
- **왜** 이 섹션이 중요한가?

```markdown
## 2. 도메인 핵심 개념
> 이 제품을 이해하려면 먼저 **도메인의 언어**를 알아야 합니다.
> 여기서는 제품에 등장하는 핵심 개념을 예시와 함께 설명합니다.
```

### 도메인 워크플로우 매핑

파이프라인 각 Step에서 도메인 워크플로우의 어느 단계에 해당하는지 명시:
```markdown
### Step 2: AI 연구설계 자동 생성
> 도메인 워크플로우 [2]~[6] — "가설부터 데이터 수집까지 자동화"
```

### 교육적 설명 기법 (도메인 개념 설명 시 필수)

- **비유 우선**: 추상적 개념은 반드시 일상 경험에 비유 후 정의
- **ASCII 시각화**: 시간축, 흐름, 계층 구조는 텍스트 다이어그램
- **관통 예시 (Running Example)**: 하나의 시나리오를 전체 문서에서 일관 사용
- **"쉬운 설명" 컬럼**: 전문 용어 표에 추가
- **"이 제품에서" 연결**: 모든 도메인 개념 마지막에 UI 연결

### 목차에 읽기 가이드

```markdown
| # | 섹션 | 내용 | 읽는 순서 가이드 |
|---|------|------|----------------|
| 1 | 제품 개요 | 제품 소개 + 도메인 배경 | 필독 — big picture |
| 2 | 도메인 핵심 개념 | 용어집 + 핵심 개념 교육 | 필독 — 모르는 용어 참조 |
| 3 | 제품 사용 흐름 | 파이프라인 Step별 설명 | 필독 — 제품 동작 이해 |
| 4 | 데이터 구조 | 엔티티 관계도 | 참조 — 구조 파악 시 |
```

## Deep Analysis (file + discussion)

Full document + follow-up AskUserQuestion 라운드로 모호한 부분 명확화.

## Incremental Analysis Protocol

분석이 여러 세션/라운드에 걸쳐 진행될 때:

### 섹션 넘버링
- 새 콘텐츠 추가 전 **현재 섹션 번호 재확인**
- 하위 섹션은 **부모 섹션 번호를 따라야** 함 (Section 3 아래는 3.1, 3.2)
- 중간 삽입 시 **이후 모든 섹션 번호 갱신**

### TOC 유지보수
- 구조 변경 후 **반드시 TOC 갱신**
- "읽는 순서 가이드" 컬럼 포함

### 상태 추적
- 문서 최상단에 `**분석 상태**: N차 완료 / 최종 완료` 명시
- 상태만 최신 유지 (변경 로그 불필요)
