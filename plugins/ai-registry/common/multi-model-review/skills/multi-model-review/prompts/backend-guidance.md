## Backend 특화 리뷰 관점

이 PR은 **백엔드 코드**를 포함합니다. 아래 6개 카테고리를 추가로 점검하세요.
발견된 이슈는 기존 카테고리(security, performance, quality, testing)로 분류하세요.

---

### 1. API 설계 → category: `quality`

| 체크포인트 | 설명 |
|-----------|------|
| REST 규약 위반 | GET 요청에서 상태 변경, 부적절한 HTTP 메서드 사용, 일관되지 않은 URL 패턴 (`/getUsers` vs `/users`) |
| 응답 형식 불일치 | 성공·실패 시 응답 구조가 다름, 페이지네이션 형식 불일치, 상태 코드 오용 (성공인데 200 아닌 코드, 에러인데 200 반환) |
| 하위 호환성 파괴 | 기존 API 필드 제거·이름 변경, 필수 파라미터 추가 — 클라이언트 배포 순서 고려 필요 |
| 입력 검증 부재 | Request body/query/path 파라미터의 타입·범위·길이 검증 없이 그대로 사용 |
| API 버저닝 | 파괴적 변경 시 버전 관리 전략 없이 기존 엔드포인트 수정 |

---

### 2. 데이터베이스 → category: `performance`

| 체크포인트 | 설명 |
|-----------|------|
| N+1 쿼리 | 루프 내 개별 쿼리 실행 — JOIN, 서브쿼리, 또는 Eager Loading으로 해결 |
| 트랜잭션 경계 | 여러 DB 작업이 원자적이어야 하는데 트랜잭션 없음 — 부분 실패 시 데이터 불일치 |
| 인덱스 부재 | WHERE/ORDER BY/JOIN 조건에 사용되는 컬럼에 인덱스 없음 — 테이블 풀 스캔 위험 |
| 마이그레이션 안전성 | 대형 테이블 ALTER, 락 발생 DDL, 롤백 불가능한 마이그레이션 — 다운타임 위험 |
| 쿼리 최적화 | SELECT * 사용, 불필요한 컬럼 로드, OFFSET 기반 페이지네이션(대량 데이터 시 성능 저하) |
| 커넥션 관리 | 커넥션 풀 미사용, 커넥션 누수(try-finally로 반환 미보장), 트랜잭션 롤백 누락 |

---

### 3. 인증/인가 → category: `security`

| 체크포인트 | 설명 |
|-----------|------|
| 미들웨어 미적용 | 새 엔드포인트에 인증 미들웨어/데코레이터 누락 — 인증 없이 접근 가능 |
| IDOR (Insecure Direct Object Reference) | 리소스 접근 시 소유권/권한 검증 없이 ID만으로 조회·수정·삭제 허용 |
| 토큰 관리 | 토큰 만료 미설정, 갱신(refresh) 로직 부재, 로그아웃 시 토큰 무효화 미처리 |
| 권한 에스컬레이션 | 일반 사용자가 관리자 기능 접근 가능한 경로 존재 — role/permission 체크 누락 |
| 비밀번호/시크릿 처리 | 평문 저장, 약한 해싱(MD5/SHA1), 시크릿의 하드코딩, 로그에 자격증명 출력 |

---

### 4. 에러 핸들링 → category: `quality`

| 체크포인트 | 설명 |
|-----------|------|
| 내부 에러 노출 | 스택 트레이스, DB 쿼리, 내부 경로가 API 응답에 포함 — 공격자에게 정보 제공 |
| 재시도/회로차단 부재 | 외부 서비스 호출 시 실패에 대한 재시도(retry with backoff), 회로차단(circuit breaker) 미구현 |
| 부분 실패 일관성 | 여러 단계 작업 중 중간 실패 시 이전 작업 롤백이나 보상 트랜잭션 미처리 |
| 에러 무시 | catch 블록에서 에러를 삼키고(swallow) 아무 처리 없이 진행 |
| 일관된 에러 형식 | 에러 응답 형식이 엔드포인트마다 다름 — 통일된 에러 응답 스키마 필요 |

---

### 5. 동시성 → category: `security`

| 체크포인트 | 설명 |
|-----------|------|
| 레이스 컨디션 | 읽기-수정-쓰기 패턴에서 동시 요청 시 데이터 덮어쓰기 — 낙관적/비관적 락 필요 |
| 데드락 위험 | 여러 리소스를 다른 순서로 잠금 획득 — 순서 보장 또는 타임아웃 필요 |
| 커넥션 풀 누수 | DB/Redis/HTTP 커넥션을 사용 후 반환하지 않음 — 풀 고갈로 서비스 중단 |
| 멱등성 미보장 | POST/PUT 요청의 재시도 시 중복 처리 발생 — 멱등성 키 또는 중복 체크 필요 |
| 공유 자원 보호 | 전역 변수, 공유 캐시, 파일 시스템 접근 시 동기화 메커니즘 부재 |

---

### 6. 로깅/관측성 → category: `quality`

| 체크포인트 | 설명 |
|-----------|------|
| 민감정보 로깅 | 비밀번호, 토큰, 개인정보(PII)가 로그에 출력 — 마스킹 또는 제외 필요 |
| 구조화 로깅 누락 | `console.log`나 문자열 연결 로깅 — JSON 구조화 로깅으로 검색·분석 용이성 확보 |
| 헬스체크 | 새 서비스/의존성 추가 시 헬스체크 엔드포인트에 반영 여부 |
| 추적 컨텍스트 | 분산 시스템에서 요청 추적을 위한 correlation ID / trace ID 전파 누락 |
| 적절한 로그 레벨 | DEBUG/INFO/WARN/ERROR 구분 없이 모든 것을 같은 레벨로 기록 |

---

> **주의**: 위 체크포인트는 추가 관점입니다. 기본 리뷰 관점(보안, 성능, 품질, 테스트)도 반드시 함께 수행하세요.
> 발견된 이슈의 `category`는 반드시 기존 4개(security, performance, quality, testing) 중 하나로 분류하세요.
