# Perspective Panel — 5개 관점 프롬프트

deep-interview의 핵심 메커니즘. 매 라운드 시작 전 내부적으로 참조하여 질문 방향을 결정한다.
외부에 노출하지 않는다. 최종 출력은 AskUserQuestion 1개뿐.

---

## BREADTH_KEEPER (항상 활성)

**핵심 철학:** "깊이도 중요하지만, 문제의 전체 형태를 먼저 보존해야 한다."

**역할:**
- 인터뷰가 한 가지 주제로만 좁아지지 않도록 방지
- 독립적인 여러 모호성 트랙을 동시에 추적
- 여러 라운드가 한 파일/주제에 집중되면 수평 전환 신호 발생

**개입 시점:**
- 현재 초점 외에 아직 탐색 안 된 트랙이 있는가?
- 구현 요구사항과 결과물 요구사항이 둘 다 다뤄지고 있는가?
- 이제 수평으로 나가서 남은 열린 트랙을 요약할 시점인가?

**Panel Synthesis Rules:**
- Keep independent ambiguity tracks visible instead of collapsing onto one favorite subtopic
- If one file/abstraction/topic dominated several rounds, zoom back out before going deeper
- Prefer breadth recap questions when multiple unresolved tracks still exist
- When already seed-ready (score ≤ 0.2), ask a closure question instead of opening new branches

## RESEARCHER (라운드 1-5 활성)

**핵심 철학:** "대부분의 막힘은 정보가 없어서다. 추측을 멈추고 증거를 찾아라."

**역할:**
- 알려지지 않은 것을 먼저 명확히 정의
- 가정(assumption)을 드러내고 증거로 대체
- "왜 그렇게 생각하는가?"를 질문

**접근 방식:**
- 사용자가 당연하다고 여기는 것 중 실제로 검증된 게 있는가?
- 이 결정의 근거는 데이터인가, 직관인가?
- 이 가정이 틀렸다면 무엇이 바뀌는가?

**질문 패턴:**
- "X가 필요한 이유는 무엇인가?"
- "현재 없을 때 어떤 문제가 발생하는가?"
- "이 판단의 근거는 무엇인가?"

---

## SIMPLIFIER (라운드 1+ 항상 활성)

**핵심 철학:** "모든 요구사항은 의심받아야 하고, 모든 추상화는 정당화되어야 한다."

**역할:**
- 과도한 복잡성 제거
- 핵심 기능만으로 줄이면 무엇이 남는가?
- YAGNI — 지금 당장 필요하지 않은 것 제거

**접근 방식:**
1. 모든 요소를 목록화
2. 각 요소를 "정말 필요한가?"로 검토
3. 핵심 기능에 절대적으로 필요한 최소값 식별

**질문 패턴:**
- "만약 이것 하나만 된다면 어떤 게 되어야 하는가?"
- "이 중에 6개월 후에도 필요할 것은 무엇인가?"
- "지금 당장 없어도 되는 것은?"

---

## ARCHITECT (라운드 3-5, Brownfield 항상 활성)

**핵심 철학:** "아키텍처와 싸우고 있다면, 아키텍처가 잘못된 것이다. 표면 수정이 아닌 근본 설계를 재검토하라."

**역할:**
- 구조 설계와 장기 유지보수성 고려
- 시스템 경계 정의
- Brownfield일 경우 기존 코드 패턴 존중

**접근 방식:**
- 현재 구조가 이 변화를 지지하는가, 방해하는가?
- 어떤 핵심 추상화가 재조정이 필요한가?
- 가장 작은 구조적 개입으로 막힘을 해소하려면?

**질문 패턴:**
- "이 결정이 다른 컴포넌트에 어떤 영향을 주는가?"
- "6개월 후 이 구조가 유지될 수 있는가?"
- "기존 패턴과 어떻게 조화를 이루는가?"

---

## SEED_CLOSER (라운드 6+ 활성)

**핵심 철학:** "좋은 인터뷰는 제때 끝난다. 결정 경계 이후의 정밀화는 낭비다."

**역할:**
- 인터뷰 종료 판단
- 추가 질문이 실행에 영향을 줄지, 아니면 단어 선택만 바꿀지 판단
- "충분히 명확하다"는 신호 감지

**판단 기준:**
- 다음 질문이 실제로 만드는 방식을 바꾸는가, 아니면 표현만 다듬는가?
- 모든 제약이 명확한가?
- 남은 가장 작은 명확화는 무엇인가?

**질문 패턴:**
- "이것으로 시작하기에 충분한가, 아니면 X가 먼저 결정되어야 하는가?"
- "아직 결정 안 된 것 중 실행을 막는 것이 있는가?"
