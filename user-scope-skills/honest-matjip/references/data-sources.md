# Data Sources — 소스별 크롤링 패턴

## 우선순위

1. **Google Maps** — 리뷰어 프로필 분석의 주력. 공개 데이터.
2. **카카오맵** — 한국 맛집 커버리지. 리뷰어 프로필은 실험적.
3. **네이버 플레이스** — 시도하되 실패 허용. 블로그 연동.
4. **WebSearch / google-search 스킬** — 전체 실패 시 fallback.

## Google Maps

### 맛집 검색
```
chromux로 Google Maps 접속
→ 검색: "[지역] [카테고리] 맛집" 또는 "[지역] restaurants"
→ 결과 리스트에서 place_id, 이름, 평점, 리뷰 수 추출
→ 상위 5~10개 후보 확보
```

### 리뷰어 프로필 접근
```
맛집 상세 페이지 → 리뷰 섹션 → 리뷰어 이름 클릭
→ /maps/contrib/[USER_ID]/reviews 페이지로 이동
→ 페이지 내용 추출: 총 리뷰 수, 레벨, 리뷰 목록
```

### 장점
- 프로필이 공개되어 있어 리뷰어 이력 분석에 최적
- anti-bot이 상대적으로 약함
- 로그인 불필요

### 주의
- 한국 로컬 커버리지가 카카오/네이버보다 부족할 수 있음
- 외국인 리뷰 포함 (필터 가능)

## 카카오맵

### 맛집 검색
```
chromux로 place.map.kakao.com 접속
→ 검색: "[지역] [카테고리]"
→ 결과 리스트에서 place_id, 이름, 평점, 리뷰 수 추출
```

### 리뷰어 프로필 접근
```
맛집 상세 페이지 → 리뷰 섹션 → 리뷰어 닉네임 클릭
→ 프로필 페이지 이동 시도
→ 성공: 리뷰 이력, 리뷰 수 추출
→ 실패: 리뷰어 이름 + 리뷰 텍스트만 기록, "프로필 미접근" 표기
```

### 주의
- 리뷰어 프로필 URL은 미문서화 — 페이지 구조가 바뀔 수 있음
- anti-bot 중간 수준 (저볼륨 20~50 req은 사례상 OK)
- 요청 간 1~2초 딜레이 권장

## 네이버 플레이스

### 맛집 검색
```
chromux로 m.place.naver.com 접속
→ 검색: "[지역] [카테고리]"
→ 결과 리스트 추출 시도
```

### Degrade Protocol (중요)
네이버는 크롤링 실패 가능성이 높다. 다음 순서로 degrade:

1. **정상 접근 성공** → 맛집 후보 + 리뷰어 프로필 분석 진행
2. **검색은 되지만 리뷰어 프로필 접근 불가** → 맛집 이름만 후보에 추가, 리뷰어 분석은 Google에서
3. **검색 자체 차단 (CAPTCHA/빈 페이지)** → 네이버 전체 포기, "네이버 접근 차단됨" 명시
4. **타임아웃** → 10초 내 로드 안 되면 포기

### 주의
- anti-bot: DataDome + 행동 분석 + IP 체크
- 해외/데이터센터 IP에서 접근 시 즉시 차단 가능
- 세션 당 재시도 최대 1회 (반복 시도하면 세션 전체 차단 위험)

## WebSearch Fallback

모든 플랫폼 크롤링 실패 시:

```
google-search 스킬 사용:
node <google-search-path>/vendor/web-search.mjs "[지역] 찐맛집 추천" --count 5

커뮤니티 소스 검색:
node <google-search-path>/vendor/web-search.mjs "[지역] 맛집" --site reddit.com --count 5
```

이 경우 리뷰어 프로필 직접 분석은 불가하므로:
- **한계를 반드시 명시**: "플랫폼 접근 제한으로 리뷰어 이력 직접 분석이 불가했습니다"
- 커뮤니티 합의(여러 사람이 추천) 기반으로 차선의 신뢰 판단
- 체험단/광고 키워드 검색은 텍스트 기반으로 가능

## 교차 등장 보너스

동일 맛집이 2개 이상 소스에 등장하면 신뢰도 가점:
- 2개 소스: +0.1
- 3개 소스: +0.2

교차 확인은 맛집 이름 + 주소 기반 매칭 (fuzzy match 허용).
